# 技术方案：公司空闲时主动找事做

## 总体形状

四个改动面，改动量从小到大排：底层代码一小块、角色配置两行、一个新技能、
章程一小段。判断力全部放在技能和章程里，底层只做"把正确的提示词发给正确的
角色"这一件事（延续"判断力=skill、底层只管接线"的一贯做法）。

```
角色 yaml 新键 idle ──→ agent_loop 读到 ──→ 空醒时渲染两种提示词之一
  (默认 stop，仅 ceo.yaml 设 proactive)
                                              ├─ stop：现状文本，逐字节不变（四个干活角色）
                                              └─ proactive：不再命令"没事就停"，
                                                 改为指向章程的空闲职责（CEO）
CEO 章程 ──→ 空醒第一步查账本 ──→ 全空才进入经营巡检 ──→ 指向新技能
新技能 when-idle ──→ 巡检怎么做：查账、冷却、开放式找事（四类举例）、硬规则、派单
```

## 改动一：底层接线（复用 `session:` 键的既有通道）

角色 yaml 增加可选键 `idle: stop | proactive`，缺省 `stop`。流水线与
`session:` 键完全同构，不发明新通道：

- `agent/spec.py`：`"idle"` 加入白名单键，`RoleSpec` 默认 `"stop"`。
- `orchestration/agent_loop.py`：`_role_config()` 多返回一项 `idle`（非法值
  WARN 后回落 `stop`，沿用 session 键的"宁可降级不可卡死"处理）；
  `agent_loop()` 和 `build_wake_prompt()` 各加一个带默认值的参数把它传进
  `_render_events()`。
- `_render_events()` 空事件分支按 idle 渲染两种文本：
  - `stop`：现状文本，**逐字节不变**（这是钉死的回归契约，四个干活角色和
    所有已归档测试都压在上面）；
  - `proactive`：大意为——"定时醒来，没有新消息。看看有没有需要处理的事：
    在跑的活儿卡住了、值得派的工作；如果公司手上一个活儿都没有，按你章程里
    的空闲职责办，不要直接停。"（英文写进代码；只定向到章程，不在提示词里
    展开机制细节——这是 issue #207 定下的"提示词只定向不披露机制"原则。）
- 必须保住的既有约束：提示词不以 `-` 开头（两种文本都以字母开头，天然满足）；
  长期目标前缀在最前、`---` 分隔（不动那段逻辑）；新会话的定向前缀组合顺序
  不变。
- `ceo.yaml` 设 `idle: proactive`。四个干活角色的 yaml 不动（吃默认值）。

不选的路线及原因：

- 全员统一改提示词文本（一行改完）——会软化干活角色的"没事就停"，而首跑
  头号败象恰恰是 growth 在空醒里自作主张，干活角色的硬停车指令是保护性的，
  不能松。
- 底层直接读账本、由代码决定升不升级——确定性更强，但把账本耦合进
  agent_loop，且用户已拍板"CEO 自己花一条命令查账本"（决策二）。

## 改动二：新技能 `when-idle`（只挂 CEO）

新目录 `agents/assets/skills/when-idle/`，`ceo.yaml` 的 skills 列表加一行。
技能正文（英文）的主干：

1. **先查账，别凭感觉**。给出确切命令（查 `$GOAL_LEDGER/*.json` 里 status
   为 open/claimed/running/verifying 的活儿数）。有活儿在跑 → 一句话停，
   这次醒来到此为止。一个都没有 → 公司在空转，往下走。
2. **冷却判断**。你是保留会话的角色，自己记得刚干过什么：如果上一轮巡检
   就在几小时内、且没有任何新信号（没有新数据、没有活儿回来、没有外部
   消息），不要重复巡检——说明现状就是"等"，写一句原因然后停。
3. **开放式找事，四类只是举例**（决策三）：真的 startup 永远有事可做。下面
   四类是常见的起点，不是清单也不排优先级——你完全可以想到别的，任何你判断
   对公司有价值的事都算数：
   - 看真实数据：产品/渠道实际跑得怎么样（`/company` 下已有的 GSC、GA4、
     GitHub 仓库现状等结论文件；缺数据本身就值得派一个"去看数据"的活儿）；
   - 改进现有资产：数据指出的短板、上次活儿留下的毛边；
   - 铺渠道推广：给已有资产多开一条渠道、多发一次推广；
   - 全新方向：走 `find-opportunity` → `decide-direction` → `objective
     propose` 全套，**不在巡检里直接换**。
4. **等待 = 旁线任务（side quest）时间**（决策四 + 首跑教训）：某个方向在等
   数据/信号，不等于公司没事干，更不是换方向的理由——绕开在等的方向，去平行
   推进别的事；想换方向永远走上面那套完整流程，不在巡检里顺手改长期目标。
5. **产出的唯一形状是派单**：巡检的产出只能是
   `python3 -m orchestration.messaging send` 派出的正式活儿——你自己不动手
   干活，也不做任何对外部世界的写操作。
6. **要压制的 LLM 默认**（三道检验的第二道，写成显式反例）：直奔新点子而
   跳过看数据；把"在等数据"当成"没事干"进而干等或推翻等待；一兴奋派一堆
   活；自己撸起袖子干而不是派单。

## 改动三：CEO 章程接线

`agents/assets/ceo-charter.md` 的 "How you decide what to pursue" 一节改三处：

- 空醒的处理顺序明确为：先查账本 → 有活儿在跑就停 → 全空进入经营巡检
  （指向 `when-idle` 技能）；
- 原有的"空醒且无方向时去设长期目标"保留，作为巡检里"全新方向"一类的
  特例（没有目标时它就是最值的事）；
- 补一句等待的定调：在等数据的方向不算无事可做——等待期是旁线任务时间，
  平行推进别的事；推翻一个写下的等待，只能因为"同一资产上出现了承诺之后
  的新信号"（首跑模式⑦的对症规则），巡检不是重开方向讨论的场合。

## 改动四：测试与收尾

- 单测（`orchestration/tests/test_agent_loop.py` + `agent/tests/`）：
  - idle 键缺省/非法值回落 `stop`；
  - `idle=stop` 空醒提示词与现状逐字节一致（回归钉）；
  - `idle=proactive` 空醒提示词含主动指引、不含"say so in one line and
    stop"；与长期目标前缀、新会话前缀的组合顺序正确；
  - `ceo.yaml` 解析出 `idle: proactive`。
- 真跑 e2e（手工，按 SOP 第 7 阶段）：起测试公司（`COMPANY=e2eidle` 之类），
  正向/反向/边界三条对应 AC2/AC3/AC4，看 telemetry 的 wake 记录、账本文件、
  hub 日志取证。
- spec 增补：`resident-agent-contracts.md` 记 idle 键契约与两种空醒文本。

## 兼容与回滚

- 全部向后兼容：idle 缺省 `stop`，不改任何 yaml 的公司行为逐字节不变；
  旧 yaml（无 idle 键）天然合法。
- 回滚点清晰：撤 `ceo.yaml` 两行（idle 键 + 技能行）即可整体退回现状，
  底层代码可以留着不撤。
